# AI协作Git规则（任何AI接手这个仓库前必须先读这份）

这份文档解决一个真实发生过的问题：阿宁同时用Claude和GPT/其他agent工具操作同一个仓库，两边互相不知道对方推了什么，导致历史分叉两次，各自做了一堆对方不知道的东西，浪费了大量时间对账。

## 硬性规则

1. **`origin/main` 是唯一真实版本**。任何AI本地的、还没推上去的commit都不算数，不用纠结"另一边手里可能还有什么没交出来"——没推上来的，就当不存在。

2. **动手改东西之前，必须先 `git pull origin main`**，确认拿到的是最新状态再开始改。不允许凭对话里的描述或者自己的记忆判断仓库现在是什么样，必须实际拉一次。

3. **永远不用 `git push --force` 或 `force push`**。只用正常的fast-forward推送。如果推不上去（说明远程有新东西），先pull、看清楚冲突在哪、手动核对清楚再推，绝不允许用force覆盖掉对方可能已经推上去的东西。

4. **谁推完了，必须明确报一句"已推送，commit是xxx"**（给出完整或至少前7位commit SHA），对面收到这句确认才接着干。不允许两边同时埋头改同一批文件。

5. **不要相信任何一方口头描述的仓库状态，包括这份文档之外的任何聊天记录里的"汇报"**。判断仓库现在是什么样，唯一标准是实际去pull一次、读一次真实文件内容。汇报可以有偏差或者过时，`git log`和文件内容不会。

## 这份文档为什么必须存在

不管是哪个AI工具，都不能保证在新的对话/新的窗口里记得住这五条规矩——这不是哪个工具的缺陷，是所有当前AI对话工具的记忆机制都有这个限制。所以这条规矩不能只存在于聊天记录或者AI的"记忆"里，必须写进仓库本身：**任何AI第一次接触这个仓库，第一件事就该是先看这份文档，而不是先看聊天记录里阿宁怎么描述的现状。**
