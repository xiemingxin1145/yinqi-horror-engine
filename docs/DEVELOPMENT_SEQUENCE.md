# 《阴契》开发顺序

## 当前判断

剧情、人物关系、村庄背景、鬼怪细节都还没有最终定稿，所以现在不能急着写正式剧情。正确做法是先做地基，让后续剧情变成可替换数据。

## 第一阶段：地基

目标：让项目不依赖固定剧情也能运行。

必须完成：

1. `EventBus`：全局事件通信。
2. `GameState`：统一存世界、玩家、NPC、线索、鬼怪、flags。
3. `DataRegistry`：统一注册地点、NPC、规则、事件模板。
4. `data/`：用 JSON 管地点、NPC、规则、事件。
5. `WorldSim`：从 DataRegistry 创建世界，不再写死归魂村。
6. `FolkloreRules`：从规则模板判定禁忌，不再手写单个案例。
7. `HorrorDirector`：从玩家状态生成事件包，并通过 EventBus 发出。
8. `Main.gd`：只演示地基，不写正式剧情。

## 第二阶段：最小可玩循环

目标：跑通一局 5 分钟原型。

循环：

```text
进入地点
→ 调查物品
→ 获得线索
→ 触发民俗规则
→ 改变风险值
→ HorrorDirector触发氛围事件
→ NPC根据记忆/恐惧改变行为
```

此阶段只需要一个小地图：

```text
村口
祠堂
老宅
河桥
戏台
```

## 第三阶段：AI NPC

目标：接 Microverse 思路。

先做：

- NPCMemory 标签记忆。
- NPCBehaviorPlanner 行为选择。
- NPC信任/恐惧/隐瞒。

后做：

- 接大模型对话。
- 接 Dialogue Manager。
- 接长期记忆库。

## 第四阶段：民俗规则库

目标：形成《阴契》自己的护城河。

第一批规则：

- 子时不可点睛。
- 祠堂不可直呼亡者名。
- 红线断不可跨门槛。
- 纸人不可背对神龛。
- 祭品不可缺三样。

每条规则必须有：

```text
触发条件
判定条件
风险变化
事件结果
线索影响
```

## 第五阶段：正式剧情

等上面地基跑通后，再写正式剧情。剧情只负责填数据，不改引擎。

例如：

```text
冥婚案 = 一组规则 + 一组NPC + 一组地点 + 一组线索 + 一个鬼怪生成参数
```

这样以后可以继续加：

```text
纸人回魂案
河神祭祀案
戏班鬼戏案
校园怪谈案
```

不推翻底层。
